### 四类思路,按干净程度排

#### 1\. Action masking(最干净,强推)

在 agent 选站时,**直接屏蔽会导致排队超时的动作** 。

实现:每个 step 给每个待决策 EV 算一个 `feasible_mask`,满足以下条件的站才可选:

- `station.queue_length × avg_service_time + dist_to_station / speed < 4h - safety_margin`
- 或更严格:预测排队时间 < 某阈值(比如 2h)

policy 根本接触不到"选爆队站"这个动作,**reward hacking 的动作空间被物理切掉** 。

**好处**:

- 不动 reward,baseline 对比公平性不破坏
- 论文里写得漂亮:"constraint-aware action space"
- Double DQN + action mask 是标配,你代码里评估时已经用过 mask,训练时接上就行

**代价**:

- 极端情况所有站都被 mask 会出问题,需要 fallback(比如全 mask 时选最近站)
- 要新加 "queue time prediction" 逻辑(但你 estimator 里其实有类似的)

#### 2\. Environment-level 硬约束(次干净)

EV 在 **决策时** 而不是 **超时后** 主动判断:"这个站我去了会超时,我不去"。

实现:EV 进入 `CHARGE_DECISION` 状态时,先过一道 feasibility 过滤,不可行的站不暴露给 agent。

和 action masking 几乎等价,区别是过滤在 env 侧做,不在 agent 侧。 **推荐用 action masking,因为它让 agent 显式知道"有动作被屏蔽",梯度计算更稳** 。

#### 3\. State 里加入排队预警信号

把"当前各站的预期排队时间 / 拥挤程度"作为 observation 的一部分。

policy 看得到信号 → 学得出回避。你现在的 state 里有没有?如果有 queue\_length,那 policy 理论上能学到,但显然没学到(因为 reward 给它的信号方向错了)。

**单独用这一条无效**,得和 reward / mask 配合。不推荐单独做。

#### 4\. Episode-level termination(粗暴但有效)

任何 EV 一旦超时放弃 → episode 立刻 done,该 episode 所有累积 reward 作废或大幅衰减。

效果强但副作用大:

- 训练早期 policy 没学好就一直被 reset,学习效率差
- 不符合真实部署(实际系统不会一辆车放弃就关机)
- 论文里不好讲

**不推荐** 。

**Claude**

这是我们加的 mask 健康度指标之一。拆开讲:

### 每次 get\_action\_mask 调用的三种结果

对某辆 EV,针对所有 8 个充电站逐个判断能不能选:

| 结果 | 含义 | 计数器 |
| --- | --- | --- |
| 全部可选 | 8 个站 mask 都是 True | total +1(不单独计) |
| **部分可选** | 部分 True,部分 False | total +1,**partial\_masked +1** |
| **全部不可选** | 8 个站 mask 都是 False | total +1,**all\_masked\_before\_fallback +1**,然后触发 fallback 强制全 True |

### all\_masked\_ratio 的定义

```
all_masked_ratio = all_masked_before_fallback / total
```

就是" **多大比例的决策场景下,所有站都被 mask 掉了** "。

### 为什么要关心它

因为这时 fallback 会触发—— `if not mask.any(): mask.fill_(True)` 强制让所有站又变可选,agent 相当于在一个 **没有任何约束** 的状态下做决策。

如果 `all_masked_ratio` 很高(比如 20%),意味着:

- 有 20% 的决策场景下,mask 说"这 8 个站都去不了"
- 然后又被 fallback 全部放开
- agent 在这 20% 场景里实际上 **没有 mask 保护**,该发生的 reward hacking 还是会发生

换句话说,**`all_masked_ratio` 越高,mask 越"形同虚设"** 。

### 健康阈值

- **< 5%**:健康。mask 在绝大多数时候真正起作用,极端场景(系统严重过载)才触发 fallback
- **5–20%**:警报。safety\_margin 可能太严,或者系统实际容量不够
- **\> 20%**:故障。mask 太严把动作空间切没了,得立刻停,要么降 safety\_margin\_h(0.5→0.3),要么检查 `estimate_queue_wait_hours` 是不是预测值虚高

### 对比 partial\_masked\_ratio

- `all_masked_ratio` 是 **坏信号** (越低越好)
- `partial_masked_ratio` 是 **好信号** (mask 在真干活,过滤了部分不合理选项)

健康画面大概是:

```
total=1200  all_masked=30 (2.5%)  partial_masked=800 (66.7%)  fallback=30
```

这种就漂亮:大部分场景下 mask 有动,但没把 agent 逼死。

### 流程拆开看

```
1. mask = get_action_mask(ev)   → 给 8 个站打 True/False 标签
     ↓  比如 [True, False, True, True, False, False, True, True]
     ↓  意思:station 1/4/5 被禁,其他能选
     
2. q_values = agent.q_network(state)   → 8 个站每个都算一个 Q 值
     ↓  比如 [-0.2, -0.1, -0.5, -0.3, -0.05, -0.4, -0.6, -0.8]
     
3. 应用 mask:把被 mask 掉的 Q 值设为 -inf
     ↓  [-0.2, -∞, -0.5, -0.3, -∞, -∞, -0.6, -0.8]
     
4. argmax:在剩下的可选站里挑 Q 值最高的
     ↓  station 0(Q=-0.2 是最大的)
```

### 关键:agent 是自己学出来选哪个,不是选"时间最少"

```

```

所以 agent 会学着选 **"用户时间又少 + 电网影响又小"** 的站——这是 **0.3:0.7 加权之后综合最优** 的站,不是单纯的最短时间。

### mask 和 Q 值的分工

|  | 职责 |
| --- | --- |
| **mask** | 硬约束,把"会导致排队超时放弃"的站砍掉(不管它们 Q 值多高) |
| **Q 值** | 软优化,在剩余可行站里挑综合 reward 最高的 |

所以正确的心智模型是:

> **"mask 负责避免灾难(abandoned),Q 值负责追求优化(voltage + user)。"**

- 约束和优化 **解耦**:不用把"别让 EV 放弃"硬塞进 reward 函数

- 审稿友好:"hard constraint + soft multi-objective reward" 是 safe RL 的常规范式,比 "penalty tuning" 更硬核
