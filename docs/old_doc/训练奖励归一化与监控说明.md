# 训练奖励归一化与监控说明

## 改动目的

针对 `train-real` 训练中出现的两个问题：

1. 训练日志中的总 reward 长期大幅波动，不容易判断模型是否真的在学习。
2. 训练用 reward 数值尺度过大，Q 学习容易不稳定。

本次改动对训练主流程做了两件事：

- 对写入 replay buffer 的 `mixed_r` 做缩放与截断
- 在训练日志中增加 reward 子项与关键代价统计

代码入口：

- `visualization/run_training_real_map.py`

## 当前 reward 处理方式

训练时实际写入经验回放池的奖励不是环境总 reward，而是单次决策的 `mixed_r`。

当前处理流程：

1. 先根据动作代价构造局部 shaped reward
2. 再叠加 realized queue / service 惩罚与 global bonus
3. 对 `mixed_r` 做缩放
4. 将缩放后的结果截断到固定范围

当前配置：

- `mixed_reward_scale = 50.0`
- `mixed_reward_min = -20.0`
- `mixed_reward_max = 5.0`

即：

```python
mixed_r = mixed_r / 50.0
mixed_r = max(-20.0, min(5.0, mixed_r))
```

## 为什么这样做

### 1. 降低 reward 量级

原始 reward 由用户成本、排队成本、电网成本等多项组成，绝对值较大。直接用于 DQN 时，容易让 Q 值和 TD loss 波动过大。

缩放后有助于：

- 减少极端负回报对训练的冲击
- 让不同轮次的训练信号更可比较
- 提高 Q 学习稳定性

### 2. 避免少量异常样本主导训练

在真实路网 + 充电排队场景下，个别极端拥堵样本会产生很大的负 reward。截断后，训练更不容易被异常 episode 拉偏。

## 新增日志项

`[Profile]` 日志中新增了以下字段：

- `avg_mixed_r`
- `avg_queue_h`
- `avg_trip_h`
- `avg_service_h`
- `avg_real_queue_h`
- `avg_real_service_h`
- `avg_cost`

含义如下：

- `avg_mixed_r`：本轮写入 replay 的平均训练 reward
- `avg_queue_h`：动作选择时预测的平均排队时长
- `avg_trip_h`：动作选择时预测的平均行驶时长
- `avg_service_h`：动作选择时预测的平均服务时长
- `avg_real_queue_h`：环境执行后的平均实际排队时长
- `avg_real_service_h`：环境执行后的平均实际服务时长
- `avg_cost`：环境执行后的平均广义成本

## 训练时建议重点观察

不要只看 `Episode Reward` 是否上升，更建议优先观察：

- `avg_mixed_r` 是否逐步稳定
- `avg_queue_h` / `avg_real_queue_h` 是否下降
- `avg_service_h` / `avg_real_service_h` 是否下降
- 后续评估中的 `avg_wait_time_min`
- 后续评估中的 `abandoned_evs`
- 后续评估中的 `evs_in_line`

## 解释建议

如果总 reward 仍有波动，但以下指标变好：

- `avg_real_queue_h` 下降
- `abandoned_evs` 下降
- `avg_wait_time_min` 下降

则通常说明模型仍在学习，只是总 reward 受多目标权衡影响较大，不适合作为唯一判断依据。

## 后续可选优化

如果后续训练仍不稳定，可继续尝试：

- 调整 `mixed_reward_scale`，例如 `30.0` 或 `100.0`
- 调整 reward 截断区间
- 固定训练 seed 做对照实验
- 周期性插入固定种子评估，而不是只看在线训练 reward
