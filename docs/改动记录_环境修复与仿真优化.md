# 改动记录：环境修复与仿真优化

## 概述

本轮对仿真环境的物理建模、时间步长、奖励函数进行了系统性修复与简化。

---

## 1. SOC 单位统一（Bug Fix）

**文件**: `env/entities.py`, `env/charging_station.py`

**问题**: `ev.soc` 使用百分比（0-100），但 `ev.target_soc = 0.95` 是小数，导致所有充电计算结果严重偏差。同时 `(ev.target_soc - ev.soc / 100.0)` 存在运算符优先级错误（先除后减）。

**改动**:
- `entities.py`: `self.target_soc = 0.95` → `self.target_soc = 95.0`
- `charging_station.py`: 4 处 `(ev.target_soc - ev.soc / 100.0)` → `(ev.target_soc - ev.soc) / 100.0`
- `charging_station.py`: `optimize_power` 中充电上界从按 100% 充满改为按 `target_soc` 计算

---

## 2. SOC 消耗改为基于行驶距离（Bug Fix）

**文件**: `env/entities.py`

**问题**: `drive_kwh_per_km` 定义了但从未使用，SOC 扣减为固定的 `2.0 * moved_hours`，与实际行驶距离无关。

**改动**:
- `__init__` 中新增 `self.current_edge_length_m = 0.0`
- `move()` 中接收 `enter_edge` 返回的 `length_m` 并存储
- SOC 消耗改为：`dist_km * drive_kwh_per_km / battery_capacity_kwh * 100.0`，距离按行驶时间在当前边上的比例折算

---

## 3. `_FlexiblePowerGrid` 重复初始化修复

**文件**: `env/power_grid.py`

**问题**: 子类手动重复了父类所有属性赋值，未调用 `super().__init__()`，父类新增字段时子类会静默缺失。

**改动**: 改为 `super().__init__()` + 覆盖 `self.lines`、`bus_voltages`、`line_losses`。

---

## 4. feat[17] 改为有效特征

**文件**: `env/base_env.py`

**问题**: 原 `feat[17]` 为 EV 是否已分配到该站的标志，几乎恒为 0，对网络无信息量。

**改动**: 改为充电站当前可用充电桩比例：
```python
float(max(0, station.num_chargers - len(station.connected_evs)) / station.num_chargers)
```

---

## 5. 简化 `estimate_average_charge_hours` 默认值

**文件**: `env/charging_station.py`

**问题**: `return max(0.25, 0.7 * self.max_charger_power / max(1e-6, self.max_charger_power))` 化简后恒为 `0.7`，写法令人困惑。

**改动**: 直接 `return 0.7`

---

## 6. 时间步长从 1 小时改为 10 分钟

**文件**: `env/base_env.py`, `env/real_env.py`, `env/charging_station.py`, `training/config.py`, `evaluation/metrics.py`

**问题**: 1 step = 1 小时过于粗糙，市区开车去充电站通常 10-30 分钟，一个 step 内就覆盖了整个充电过程，Agent 无法学习动态调整。

**改动**:
- `base_env.py`: `step_duration_h = 1/6`, `steps_per_day = 144`
- `real_env.py`: `__init__` 和 `reset()` 中同步修改（原来硬编码覆盖了基类设置）
- `charging_station.py`: `step()` 新增 `step_duration_h` 参数，`energy_kwh = power * step_duration_h`，`total_fee_paid` 计费也乘以 `step_duration_h`
- `base_env.py`: 调用 `station.step()` 时传入 `step_duration_h`；`grid_cost` 中乘以 `step_duration_h` 将功率换算为能量
- `config.py`: 所有档位 `steps_per_episode` × 6，保持每 episode 覆盖的仿真时长不变
- `metrics.py`: `CostParams.STEP_DURATION_H = 1/6`；`grid_cost` 累计时乘以 `STEP_DURATION_H` 将 kW 转换为 kWh

## 7. 路网扩大：从珠江新城改为广州天河区

**文件**: `download_map.py`（新建）, `training/config.py`, `training/trainer.py`, `evaluation/run_evaluation.py`

**问题**: 珠江新城路网较小，策略间对比效果不明显；训练时 `max_nodes=200` 过小，路由决策缺乏复杂性。

**改动**:
- 新建 `download_map.py`，下载 `Tianhe District, Guangzhou, China` 路网，保存为 `guangzhou_tianhe.graphml`（支持多 Overpass 服务器自动切换）
- `config.py`: `_DEFAULT_GRAPHML` 改为 `guangzhou_tianhe.graphml`，`max_nodes: 200 → 800`
- `trainer.py`, `run_evaluation.py`: 统一替换旧路网文件名引用

## 8. 奖励函数简化

**文件**: `training/trainer.py`

**问题**: 原 reward 包含 10+ 个加权项，`generalized_cost` 已包含 queue/trip time 但二者又被单独叠加（重复计数），事前估计与事后实现成本混用，权重无物理依据，梯度方向不稳定。

**改动**: 简化为单一目标——最小化等待时间 + 行驶时间：

```python
# 改前（7项叠加）
per_ev_r = -generalized_cost - 18*queue_h - 8*service_h - 1*trip_h - 3*grid_load_ratio
mixed_r = per_ev_r - 0.2*realized_cost - 10*realized_queue_h - 4*realized_service_h + global_bonus
mixed_r = mixed_r / 50.0; clamp(-20, 5)

# 改后（1项）
per_ev_r = -(queue_time_h + trip_time_h)
mixed_r = per_ev_r / 2.0; clamp(-1.0, 0.0)
```

`mixed_r ∈ [-1, 0]`，0 为理想（零等待零行驶），-1 对应等待+行驶合计 2 小时。
电网侧约束（电压、功率波动）保留在 env reward 中，不混入 per-EV 训练信号。
